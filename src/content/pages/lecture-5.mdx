A prediction is only useful if someone acts on it. A churn score of 0.87 means nothing to a marketing manager who needs to decide what to do about it. This lecture wraps the prediction API with an LLM so raw scores become explanations, suggested actions, and interactive Q&A.

But an LLM in production is a new source of risk. It can fabricate facts, ignore instructions, and be manipulated through its input. We treat these as instances of the same problem we raised in L1: the system will produce wrong outputs, and the design must handle that. Guardrails, grounding in real data, fallback to the model's predicted values when the LLM output cannot be verified — these are the system-level answers to the LLM's tendency to sound confident while being wrong.

We also touch on the broader ML security landscape beyond prompt injection: adversarial inputs designed to fool the model, poisoned training data, and model extraction attacks. These are not theoretical — they have been demonstrated against deployed systems.

Students build the hybrid architecture that defines modern AI products: a prediction engine with a conversational front-end. The integration is straightforward. Making it safe enough to trust is not.
